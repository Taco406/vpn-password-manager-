# Publishing the SENTINEL extension to the Chrome Web Store

Today the browser extension installs via **Load unpacked** (see [`setup.md`](./setup.md) ¬ß3), which
works but requires Developer mode and an occasional browser nag. Publishing to the Chrome Web Store
gives users a **one-click install** with no Developer-mode requirement. This is optional and needs
**your** Google account ‚Äî Claude can prepare everything, but only you can submit under your account.

## What it costs / requires

- A **Google account** and a one-time **$5** developer-registration fee.
- A review by Google (typically a few days for a small extension).

## Step 1 ‚Äî Register (one time)

1. Go to the [Chrome Web Store Developer Dashboard](https://chrome.google.com/webstore/devconsole).
2. Sign in and pay the one-time **$5** registration fee.

## Step 2 ‚Äî Build the upload package

From a checkout with the toolchain (or ask Claude to produce the zip):

```
pnpm --filter @sentinel/extension build
cd apps/extension/dist && zip -r ../sentinel-extension.zip . && cd -
```

This produces `apps/extension/sentinel-extension.zip` containing `manifest.json`, `background.js`,
`content.js`, `popup.html`, `popup.js`.

> **Note on the extension ID.** The unpacked build has a fixed id
> (`pbcngnmfielibgghcofedjmojogohcdf`) from the pinned `key` in `manifest.json`. The Web Store
> assigns its **own** id on publish. See Step 5 ‚Äî the app must allow *both*.

## Step 3 ‚Äî Create the listing

In the dashboard: **Add new item** ‚Üí upload `sentinel-extension.zip`, then fill:

- **Description** ‚Äî see the copy in [`store-listing.md`](./store-listing.md).
- **Category** ‚Äî Productivity.
- **Screenshots** ‚Äî reuse the app screenshots in `showcase/` (generated by `pnpm screenshots`).
- **Privacy** ‚Äî link the policy at [`privacy-policy.md`](./privacy-policy.md) (host it, e.g. as a
  GitHub Pages page or in the repo). Declare **no data collection** ‚Äî the extension sends nothing to
  any server; it only talks to the local SENTINEL app over native messaging.
- **Permissions justification** ‚Äî `nativeMessaging` (talk to the local app), `activeTab` +
  `clipboardWrite` (fill/copy on the current tab), broad host access (autofill can be offered on any
  site; a site only ever receives credentials you saved for it).

## Step 4 ‚Äî Submit for review

Submit and wait for approval. You'll get an email when it's live, and the store URL + final
extension id.

## Step 5 ‚Äî Wire the store id into the app (Claude does this)

Once published, send Claude the store extension id. Claude sets `STORE_EXTENSION_ID` in
`apps/desktop/src-tauri/src/nmhost.rs` (currently `None`) so the native-messaging host allows **both**
the unpacked-dev id and the store id, and ships it in the next release. After that, store-installed
users need no Developer mode at all.
